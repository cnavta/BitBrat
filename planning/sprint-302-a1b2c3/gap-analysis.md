# Gap Analysis: Adventure System Phase 1 (MVP)

This document analyzes the discrepancies between the requirements defined in `architecture.yaml` (and the updated architecture brief) and the actual implementation as of Sprint 302.

## 1. Persistence Feedback Loop (Major Gap)

**Requirement:** The system should maintain a persistent world-state and historical choice tracking. The LLM narrates scenes and provides choices, which are then used by the user.

**Implementation Reality:**
- `story-engine-mcp.ts` provides `process_action` to record user input.
- However, there is **no tool** for the LLM to save its narrated scenes and generated choices back to the persistence layer.
- The `adventure-tools.ts` only provides read/initialization tools.
- Result: Scenes generated by the LLM are volatile and exist only in the chat history, making long-term state tracking and "Save/Load" functionality impossible without parsing old chat logs.

## 2. Stability of `get_current_scene` (Critical Bug)

**Requirement:** `get_current_scene` should return the current state of the adventure.

**Implementation Reality:**
- In `src/apps/story-engine-mcp.ts`, the `get_current_scene` function retrieves the last entry from the history.
- If the last entry is a `user_action` (added via `process_action`), the code attempts to access `entry.scene` and `entry.choices`.
- Since `user_action` entries do not contain these fields (they only have `action`), the service will crash or return `undefined` for critical narrative data.
- **Root Cause:** The logic assumes every history entry is a narrated scene, but user actions are also appended to history.

## 3. Session Resumption (Functional Gap)

**Requirement:** Resilient and persistent adventure sessions.

**Implementation Reality:**
- `llm-bot` lacks the logic to bootstrap its context from the `story-engine-mcp` when an adventure is resumed after a bot restart or a long pause.
- The bot relies on ephemeral in-process memory or the immediate chat context.
- If the bot's memory is cleared, it has no way to "ask" the story engine for the full narrative summary to rebuild its internal state.

## 4. `start_story` API Discrepancy (Minor)

**Requirement:** The execution flow suggests `start_story` should return the first scene to jumpstart the adventure.

**Implementation Reality:**
- `story-engine-mcp:start_story` currently returns a simple string: `"Story started with theme: [theme]"`.
- This requires the `llm-bot` to make an additional call to generate the first scene, or it results in a "cold start" where the user has to wait for a second interaction.

## 5. Auditability

**Requirement:** Voting results and snapshots should be persisted for auditability.

**Implementation Reality:**
- While Firestore is used, the snapshotting mechanism for the *narrative state* (not just the poll results) is not explicitly implemented in the MCP server.

---
**Summary of Findings:**
The core adventure logic is currently "write-only" for user actions and "read-only" for the LLM. To achieve Phase 1 parity, a feedback loop allowing the LLM to commit scenes to the story history is required.
