# Tutorial: Creating a Domain MCP Server

In this tutorial, you will learn how to create a **Domain MCP Server** — a Domain Bit that exposes custom MCP tools for extending the BitBrat platform with domain-specific capabilities.

## What You'll Build

We'll create a simple **weather service** that:
- Exposes MCP tools (`weather.current`, `weather.forecast`)
- Can be called by LLM agents or reflexes
- Follows the Bit model and Platform Ring conventions
- Uses the `mcp-server` profile with `platform+domain` exposure

## Prerequisites

- Platform is running locally (see [Quickstart](../getting-started/quickstart.md))
- Basic understanding of [The Bit Model](../concepts/bit-model.md)
- Familiarity with [Capability Profiles](../concepts/capability-profiles.md)
- Node.js and TypeScript development experience

## Understanding Domain MCP Servers

Before we begin, let's understand Domain MCP Servers:

| Concept | Value |
|---------|-------|
| **Category** | `domain` (optional extension, not core orchestration) |
| **Profile** | `mcp-server` (serves MCP tools) |
| **Exposure** | `platform+domain` (required for mcp-server profile) |
| **Purpose** | Provide domain-specific tools callable by LLM agents and reflexes |
| **Examples** | `obs-mcp` (OBS control), `image-gen-mcp` (image generation), `story-engine-mcp` (storytelling) |

**Key Insight:** Domain MCP Servers extend the platform with optional capabilities. They can be removed without breaking the agent loop.

See [Choosing Platform vs Domain](../guides/choosing-platform-vs-domain.md) for the decision framework.

## Step 1: Generate the Bit Scaffold

Use the `brat bit create` command to generate the service scaffold:

```bash
npm run brat -- bit create weather-mcp \
  --category domain \
  --profile mcp-server \
  --port 3030 \
  --register \
  --active
```

**What this does:**
- Creates `src/apps/weather-mcp-service.ts` (service implementation)
- Creates `src/apps/weather-mcp-service.test.ts` (test file)
- Creates `Dockerfile.weather-mcp` (multi-stage build)
- Creates `infrastructure/docker-compose/services/weather-mcp.compose.yaml`
- Registers the service in `architecture.yaml` with:
  - `category: domain`
  - `profile: mcp-server`
  - `mcp.exposure: platform+domain` (automatically set for mcp-server)
  - `active: true` (deployable)
  - `port: 3030`

## Step 2: Understand the Generated Code

The generated `weather-mcp-service.ts` will look like this:

```typescript
import { z } from 'zod';
import { Bit } from '../common/base-server';

/**
 * WeatherMcpServer
 * Generated by brat bit create
 * Profile: mcp-server
 * MCP Exposure: platform+domain
 * Kind: mcp-server
 */
export class WeatherMcpServer extends Bit {
  constructor() {
    super({
      mcpExposure: 'platform+domain',
    });
    this.registerDomainTools();
  }

  async initialize(): Promise<void> {
    const logger = this.getLogger();
    logger.info('weather-mcp.initialize.start');
    // Add initialization logic here
    logger.info('weather-mcp.initialize.complete');
  }

  private registerDomainTools() {
    // TODO: Register your domain tools here
  }
}

if (require.main === module) {
  const server = new WeatherMcpServer();
  const port = parseInt(process.env.PORT || '3030', 10);

  server
    .initialize()
    .then(() => server.start(port))
    .catch((err) => {
      server.getLogger().error('weather-mcp.startup.failed', {
        error: err instanceof Error ? err.message : String(err)
      });
      process.exit(1);
    });
}
```

**Key points:**
- Extends `Bit` base class (gets Platform Ring automatically)
- `mcpExposure: 'platform+domain'` exposes both `bit.*` and domain tools
- `registerDomainTools()` is where you register your custom tools
- `initialize()` runs before server starts (for setup tasks)

## Step 3: Implement Domain Tools

Now let's implement the weather tools. We'll use a simple mock implementation (you would integrate with a real weather API in production):

```typescript
private registerDomainTools() {
  // Tool 1: Get current weather
  this.registerTool(
    'weather.current',
    'Get current weather for a location',
    z.object({
      location: z.string().describe('City name or coordinates (e.g., "San Francisco" or "37.7749,-122.4194")'),
      units: z.enum(['imperial', 'metric']).optional().default('imperial').describe('Temperature units'),
    }),
    async (args) => {
      try {
        const logger = this.getLogger();
        logger.info('weather.current.request', { location: args.location, units: args.units });

        // Mock implementation - replace with real API call
        const temp = args.units === 'metric' ? 22 : 72;
        const unit = args.units === 'metric' ? '°C' : '°F';

        const weather = {
          location: args.location,
          temperature: temp,
          unit: unit,
          condition: 'Partly Cloudy',
          humidity: 65,
          windSpeed: 10,
          timestamp: new Date().toISOString(),
        };

        logger.info('weather.current.success', { location: args.location });

        return {
          content: [{
            type: 'text',
            text: `Current weather in ${weather.location}:\n` +
                  `Temperature: ${weather.temperature}${weather.unit}\n` +
                  `Condition: ${weather.condition}\n` +
                  `Humidity: ${weather.humidity}%\n` +
                  `Wind Speed: ${weather.windSpeed} mph`,
          }],
        };
      } catch (error) {
        return {
          content: [{
            type: 'text',
            text: `Failed to get weather: ${error instanceof Error ? error.message : String(error)}`,
          }],
          isError: true,
        };
      }
    }
  );

  // Tool 2: Get weather forecast
  this.registerTool(
    'weather.forecast',
    'Get weather forecast for a location',
    z.object({
      location: z.string().describe('City name or coordinates'),
      days: z.number().int().min(1).max(7).optional().default(3).describe('Number of forecast days (1-7)'),
      units: z.enum(['imperial', 'metric']).optional().default('imperial'),
    }),
    async (args) => {
      try {
        const logger = this.getLogger();
        logger.info('weather.forecast.request', {
          location: args.location,
          days: args.days,
          units: args.units
        });

        // Mock implementation - replace with real API call
        const forecasts = [];
        for (let i = 0; i < args.days; i++) {
          const temp = args.units === 'metric' ? 20 + i : 68 + i * 2;
          const unit = args.units === 'metric' ? '°C' : '°F';
          forecasts.push({
            day: i + 1,
            temperature: temp,
            unit: unit,
            condition: i % 2 === 0 ? 'Sunny' : 'Cloudy',
          });
        }

        const forecastText = forecasts
          .map(f => `Day ${f.day}: ${f.temperature}${f.unit}, ${f.condition}`)
          .join('\n');

        logger.info('weather.forecast.success', {
          location: args.location,
          days: forecasts.length
        });

        return {
          content: [{
            type: 'text',
            text: `${args.days}-day forecast for ${args.location}:\n\n${forecastText}`,
          }],
        };
      } catch (error) {
        return {
          content: [{
            type: 'text',
            text: `Failed to get forecast: ${error instanceof Error ? error.message : String(error)}`,
          }],
          isError: true,
        };
      }
    }
  );
}
```

**Tool registration pattern:**
```typescript
this.registerTool(
  'tool.name',              // Tool name (domain.action convention)
  'Description',            // Human-readable description
  zodSchema,                // Zod schema for parameter validation
  async (args) => { ... }   // Handler function
);
```

## Step 4: Add Configuration (Optional)

If your service needs API keys or configuration, add them to `architecture.yaml`:

```yaml
services:
  weather-mcp:
    category: domain
    profile: mcp-server
    active: true
    port: 3030
    entry: src/apps/weather-mcp-service.ts
    mcp:
      exposure: platform+domain
    env:
      WEATHER_API_URL: https://api.weather.example.com
    secrets:
      - WEATHER_API_KEY  # Resolved from Secret Manager
```

Access configuration in your code:

```typescript
async initialize(): Promise<void> {
  const logger = this.getLogger();
  logger.info('weather-mcp.initialize.start');

  // Get configuration
  const apiUrl = this.getConfig('WEATHER_API_URL');
  const apiKey = this.getSecret('WEATHER_API_KEY');

  logger.info('weather-mcp.initialize.config', { apiUrl });
  logger.info('weather-mcp.initialize.complete');
}
```

  "name": "weather-command",

### Resource Management

If your service needs to manage resources (databases, connections):

```typescript
async initialize(): Promise<void> {
  // Get persistence resource (PostgreSQL or Firestore legacy)
  const persistence = this.getResource('persistence');

  // Get Publisher resource
  const publisher = this.getResource('publisher');

  // Custom resource initialization
  this.weatherCache = new WeatherCache();
}

async close(reason: string): Promise<void> {
  // Clean up resources
  if (this.weatherCache) {
    this.weatherCache.close();
  }

  await super.close(reason);
}
```

### Tool Metrics

Expose custom metrics endpoint:

```typescript
constructor() {
  super({
    mcpExposure: 'platform+domain',
    setup: (app) => {
      app.get('/metrics', (req, res) => {
        res.set('Content-Type', 'text/plain');
        res.send(this.getMetrics());
      });
    },
  });
}

private getMetrics(): string {
  return `# HELP weather_requests_total Total weather requests
# TYPE weather_requests_total counter
weather_requests_total{tool="current"} ${this.currentRequests}
weather_requests_total{tool="forecast"} ${this.forecastRequests}
`;
}
```

## Best Practices

1. **Use descriptive tool names**: Follow `domain.action` convention (e.g., `weather.current`, `obs.set-scene`)
2. **Validate inputs with Zod**: Leverage Zod's type safety and validation
3. **Return consistent formats**: Always return `{ content: [...] }` with optional `isError`
4. **Log with correlation IDs**: Use structured logging for traceability
5. **Handle errors gracefully**: Catch and return error messages, don't throw
6. **Set appropriate timeouts**: Default is 5000ms; adjust for slow operations
7. **Document your tools**: Include clear descriptions and parameter docs
8. **Test thoroughly**: Write unit tests and integration tests
9. **Monitor performance**: Expose metrics for observability
10. **Use environment config**: Never hardcode secrets or URLs

## Common Patterns

### External API Integration

```typescript
private async callWeatherAPI(location: string): Promise<WeatherData> {
  const apiUrl = this.getConfig('WEATHER_API_URL');
  const apiKey = this.getSecret('WEATHER_API_KEY');

  const response = await fetch(`${apiUrl}/current?q=${location}`, {
    headers: { 'Authorization': `Bearer ${apiKey}` },
  });

  if (!response.ok) {
    throw new Error(`Weather API error: ${response.statusText}`);
  }

  return response.json();
}
```

### Caching

```typescript
private cache = new Map<string, { data: any; expiry: number }>();

private getCached(key: string): any | null {
  const entry = this.cache.get(key);
  if (!entry || Date.now() > entry.expiry) {
    this.cache.delete(key);
    return null;
  }
  return entry.data;
}

private setCache(key: string, data: any, ttlMs: number = 300000): void {
  this.cache.set(key, { data, expiry: Date.now() + ttlMs });
}
```

### Database Integration

**PostgreSQL (Default)**:
```typescript
import type { DocumentStore } from '../common/persistence/document-store';

async initialize(): Promise<void> {
  const persistence = this.getResource('persistence');
  this.db = persistence as DocumentStore;
}

private async saveWeatherData(location: string, data: WeatherData): Promise<void> {
  await this.db.set(`weather/${location}`, {
    ...data,
    timestamp: new Date(),
  });
}
```

**Firestore (Legacy)**:
```typescript
import type { FirestoreResource } from '../common/resources/firestore-manager';

async initialize(): Promise<void> {
  const firestore = this.getResource<FirestoreResource>('firestore');
  this.db = firestore.firestore();
}

private async saveWeatherData(location: string, data: WeatherData): Promise<void> {
  await this.db.collection('weather').doc(location).set({
    ...data,
    timestamp: new Date(),
  });
}
```

## Troubleshooting

### Tool Not Discovered

1. **Check exposure**: Is `mcp.exposure: platform+domain` set in architecture.yaml?
2. **Check registration**: Did you call `this.registerTool()` in `registerDomainTools()`?
3. **Check tool-gateway**: Is the tool-gateway receiving registration events?
4. **Check logs**: Look for `mcp.registration.published` in service logs

### Tool Execution Fails

1. **Check tool name**: Ensure sanitized name is correct (use brat fleet info to verify)
2. **Check parameters**: Validate Zod schema matches input
3. **Check timeout**: Is tool execution exceeding timeout (default 5000ms)?
4. **Check logs**: Look for error messages in service logs

### Service Won't Start

1. **Check port conflicts**: Is port 3030 (or your configured port) already in use?
2. **Check environment**: Are required env vars set?
3. **Check secrets**: Are secrets configured in Secret Manager?
4. **Check logs**: Look for startup errors in docker logs

## Summary

You've created a Domain MCP Server that exposes custom tools to the BitBrat platform! Domain MCP Servers are powerful extension points that allow you to add any capability you need.

**Key takeaways:**
- Domain MCP Servers use `category: domain` + `profile: mcp-server` + `exposure: platform+domain`
- Tools are registered with Zod schemas for type safety
- Tools are callable by LLM agents, reflexes, and other Bits
- Platform Ring (`bit.*`) is automatically included
- Services follow the Bit model and get uniform administration via `brat fleet`
